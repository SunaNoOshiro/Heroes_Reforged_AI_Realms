## 2. Critical Issues

### C1. Registry parser bleeds markdown body into `estimatedTime`
- 6 tasks have `estimatedTime` field of the form `"- 4 hours --- ## Worked Example: ..."`.
- Affected (verified):
  - `mvp.05-adventure-map.01-strategic-game-state-model`
  - `mvp.05-adventure-map.03-hero-movement`
  - `mvp.10-heuristic-ai.05-difficulty-levels-pawn-and-knight`
  - `phase-2.01-spells-artifacts.00-hero-leveling`
  - `phase-2.01-spells-artifacts.01-hero-skill-assignment`
  - `phase-2.01-spells-artifacts.04-baseline-skill-pack`
- Cause: `parseSections()` in [scripts/generate-task-registry.mjs:70-87](../../scripts/generate-task-registry.mjs#L70-L87) keeps reading until the next labeled section header (or EOF). It does not stop at the first `---` rule or `## Heading`.
- Effect: `estimatedTime` parses as a string but consumers reading `parseFloat(estimate)` survive only by luck. Anything that displays the field shows a wall of text. AI agents treating the registry as canonical will get noisy time budgets.
- Severity: **medium** — does not break the lint/cycle/dep validators, but corrupts a registry field downstream.

### C2. Phase-2 UI-screen-backlog tasks delegate command discovery to screen packages
- 51 tasks under `phase-2.07-ui-screen-backlog` use the same template: "every command listed in `<screen>/interactions.md` is dispatched, guarded, or explicitly disabled."
- Screen specs collectively reference **239** unique `ALL_CAPS` command-shaped tokens. The engine task backlog explicitly defines roughly **13 gameplay commands**: `MOVE_HERO`, `RECRUIT_UNITS`, `BUILD_BUILDING`, `BATTLE_ATTACK/WAIT/DEFEND/MOVE/RETREAT`, `INIT_BATTLE`, `SPELL_CAST`, `LEARN_SPELL`, `LEVEL_UP`, `ASSIGN_SKILL`, plus the 14 in `command.schema.json`.
- An AI agent picking a screen task will read commands like `TRADE_RESOURCES`, `HIRE_TAVERN_HERO`, `BUY_ARTIFACT`, `BUY_WAR_MACHINE`, `LEARN_UNIVERSITY_SKILL`, `RELEASE_PRISON_HERO`, `RECRUIT_EXTERNAL_DWELLING_UNITS`, `RETREAT_BEFORE_BATTLE`, `ACCEPT_BATTLE_SURRENDER`, `BUILD_BOAT`, `BUILD_GRAIL_STRUCTURE`, `GENERATE_RANDOM_MAP`, `ROLL_RMG_SEED`, `EQUIP_HERO_ARTIFACT`, `LOAD_GAME_SLOT`, `APPLY_OPTIONS`, `ACKNOWLEDGE_CALENDAR_ANNOUNCEMENT`, `COMPLETE_CINEMATIC` — and have no engine task to dispatch them against.
- Severity: **high** for autonomous execution of phase-2 — the agent would either invent reducers (drift), or implement screens against missing handlers (broken acceptance).

### C3. `command.schema.json` is the only canonical command list and lags far behind
- `content-schema/schemas/command.schema.json` declares **14** commands. The engine task backlog adds ~7 more. UI screen specs reference **239**.
- There is no engine task that owns "expand command schema to cover X system" beyond the per-feature seeds. The backlog has no `add TRADE_RESOURCES command`, `add HIRE_TAVERN_HERO command`, `add adventure-map random map generator commands`, etc.
- Severity: **high** — same root cause as C2, but it surfaces at the schema layer too.

## 5. Task Rewrites

### R1. Phase-2.07 UI-screen-backlog tasks (51 tasks) need engine-side anchors

These tasks are well-formed UI tasks but fail dimension #6 (AI-executability) because the agent cannot dispatch a command that has no reducer. Two options:

**Option A — Pair-task model (preferred).** Split each screen into:
- `screen-<n>-engine-commands.md` (engine-side: reducers, validators, state slice)
- `screen-<n>-ui-implementation.md` (UI-side: today's task, depends on the engine pair)

**Option B — Tighter scope.** Rewrite the boilerplate template:
```
Acceptance Criteria:
- Layout matches mockup.
- For each command in interactions.md:
  - if the command exists in engine command registry, dispatch it
  - if not, render the control as disabled with localized "not yet implemented" tooltip
  - the task is allowed to add a stub command to `src/engine/commands/<name>.ts`
    that returns `ValidationError("STUB")` only if a corresponding engine task does not yet exist
- Document any added stub commands in `docs/architecture/command-schema.md` follow-ups list.
```
Option B keeps the 51 tasks at ~3h each and makes them safely AI-executable. Option A grows the backlog by ~50 tasks but produces a real game.

### R2. The 6 tasks bleeding "Worked Example" content into `estimatedTime`

Move the worked-example sections behind a recognized section header so the parser stops cleanly. Either:
- Add `Worked Example:` to the parser's section list (preferred — preserves examples), or
- Wrap them as `<!-- worked-example -->` HTML comments / move to a sibling `*-examples.md`.

Concrete fix: amend `parseSections()` in [scripts/generate-task-registry.mjs:70-87](../../scripts/generate-task-registry.mjs#L70-L87) to also stop at:
- a line of three or more dashes (`^---+\s*$`), and
- any markdown heading line (`^#{1,6}\s`).

### R3. Numbering collisions in `phase-2.01-spells-artifacts/`

Two pairs of files share number prefixes (different unique IDs, but visually confusing):
- `01-hero-skill-assignment.md` + `01-spell-school-loader-plus-mastery-scaling.md`
- `04-baseline-skill-pack.md` + `04-mage-guild-content-level-15-random-selection.md`

Rewrite to alphabetical suffixes per the README rule (e.g. `01a`, `01b`, `04a`, `04b`).

### R4. Path-ownership overlaps

14 paths have multiple "owners" in the registry. Most are intentional (different mod packs reusing relative names like `manifest.json`, `faction.json`, `units/`). Real concerns:

- `src/ai/bots/difficulty.ts` — owned by 3 tasks across MVP / phase-2 / phase-3 (additive difficulty levels). Rewrite each acceptance criterion to "extend the existing module without breaking earlier difficulty assertions" and gate with a regression test.
- `src/ai/bots/tactical-evaluator.ts` — owned by `mvp.10` and `phase-2.02`. Same pattern.
- `src/ui/components/SaveLoadScreen.tsx` — owned by both save/load UI and export/import JSON. Either merge or split the file.
- `src/engine/artifacts.ts` — owned by paper-doll task and combination-artifacts task. Merge or split the file.
- `baseline.ruleset.json` — relative-path collision; pin to `resources/packs/baseline-ruleset/ruleset.json` everywhere to disambiguate.
