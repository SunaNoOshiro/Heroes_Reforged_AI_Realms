# Heroes Reforged: AI Realms — Full Repository Audit

Date: 2026-04-20
Scope: design, schemas, tasks, docs. No runtime code exists yet (every
`src/*` folder is a README stub), so this audit grades the plan, the
contracts, and the paper architecture only.

---

## 1. Overall Scores (0–10)

| Dimension | Score | One-line verdict |
|---|---|---|
| Architecture (design) | 8.0 | Clean separation; determinism stack well-defined; a few contradictions between docs and examples. |
| Game Systems (specified) | 6.0 | Combat math is specified but leaks floats into a "no floats" codebase; spell/ability/artifact contracts are almost empty. |
| AI Generation Readiness | 4.5 | Schemas too permissive where it matters; no effect registry; no prompt-to-JSON contract; auto-balance target unrealistic. |
| Backlog Quality | 7.5 | Good sizing (2–6h), good dependency labeling, clear outputs. Weak cross-consistency with the schemas they compile against. |
| AI-Friendliness of Repo | 7.0 | Great for navigation and picking tasks; weak for executing content generation. |

Weighted verdict: ~6.5/10. A strong architectural skeleton, but the
content contracts and AI-generation surface — which are the
differentiators of this project — are the thinnest part.

## 2. Critical Issues (must fix)

### C1. `Castle` vs `Emberwild` split-brain
- `docs/architecture/pack-contract.md` names `castle-faction`,
  `necropolis-faction`, `legacy-library`, `legacy-ruleset`.
- `tasks/mvp/04-faction-castle.md` says Castle is the reference faction.
- `tasks/phase-3/02-ai-generation/03-auto-balancer-headless-battle-baseline.md`
  balances vs Castle.
- But the only example pack in `content-schema/examples/packs/` is
  Emberwild.

Single most confusing thing in the repo. Pick one "first faction" name
and align docs, examples, tasks, and pack folders. Preferred fix:
rename tasks to `04-faction-emberwild*` or relabel Emberwild as
example-only and add a real Castle example.

### C2. Fixed-point rule is violated by the schemas themselves
`docs/architecture/overview.md` forbids floats in deterministic paths.
But `content-schema/schemas/unit.schema.json` declares `attack`,
`defense`, `hp`, `speed`, `damageMin/Max` as `"type": "number"` (any
JSON number, including floats). And the legacy expansion damage task
(`tasks/mvp/09-tactical-combat/03-damage-formula-legacy-accurate.md`) uses
`0.05`, `0.025`, `×2.4`, `1/1.4` literals.

Fix: change stat fields to `"type": "integer"` (or a documented
milliunit integer), and restate the damage formula in ruleset constants
as fixed-point ratios (e.g. `atkBonusPerPointNum: 1,
atkBonusPerPointDen: 20` → 1/20 == 0.05). Otherwise the first real
commit breaks the determinism non-negotiable.

### C3. Ruleset formulas are strings with no evaluator contract
`content-schema/schemas/ruleset.schema.json` stores formulas as
free-form strings. No DSL is defined, no evaluator is planned in any
task file. Two bad endpoints:

- If evaluated via `new Function(...)`/`eval`: breaks sandbox policy
  for user/AI packs, breaks determinism portability (JS float math).
- If evaluated as a custom mini-DSL: nobody has designed that DSL, and
  M3 spell/artifact work assumes it works.

Fix: add an explicit task under `02-content-schemas/` titled "Formula
DSL grammar + evaluator", define allowed opcodes (add/sub/mul/div-fixed,
min/max, clamp, lookups), and reference it from the ruleset schema. Or
change formulas from strings to structured AST JSON.

### C4. Spell/Ability/Artifact effect contracts are effectively unvalidated
- `content-schema/schemas/spell.schema.json` — `targeting`,
  `masteryTiers`, `presentation` all `additionalProperties: true`; root
  also `additionalProperties: true`.
- `content-schema/schemas/ability.schema.json` — effects only require
  `type: string`; everything else permitted.
- `content-schema/schemas/artifact.schema.json` — `effects`, `combo`,
  `restrictions`, root all `additionalProperties: true`.

An AI-generated spell pack will pass validation while being
semantically garbage. This is the single biggest blocker for AI
generation readiness.

Fix: introduce an effect-kind registry (enum of permitted `type`
values, one subschema per kind using `oneOf`/discriminator). Do the
same for `targeting.kind` and `restrictions.kind`. Lock
`additionalProperties: false` on the root once the enum is stable.

### C5. No content-hash pinning in the manifest
`content-schema/schemas/manifest.schema.json` has no `contentHash` or
`lock` field, yet `docs/architecture/content-platform.md` says "Saves,
replays, and multiplayer pin versions and content hashes." Phase-3
multiplayer `04-per-turn-hash-exchange-plus-desync-detection.md` will
silently lack the pinning primitive.

Fix: add `contentHash` (hex string, canonical-JSON-of-records hash) to
manifest, and an `engineHash` field; have `src/content-runtime/` emit
it at pack build time.

### C6. Provider-neutral AI boundary has no I/O contract
`docs/architecture/ai-integration.md` describes interfaces in prose
only. There is no `content-schema/schemas/faction-generation-request.schema.json`,
no output schema for `RawFactionData`, no prompt template fixture. The
phase-3 tasks assume this boundary works. Add:

- `content-schema/schemas/generation-request.schema.json`
- `content-schema/schemas/generated-faction.schema.json`
- an example prompt fixture under `content-schema/examples/generation/`.

## 3. Important Improvements

### I1. Missing record types in schema-matrix but referenced elsewhere
- `Skill` is referenced (`faction.skillIds`, `hero.startingSkillIds`)
  but has no schema.
- `Scenario`, `HeroClass`, `TerrainSet`, `Generator preset` are
  referenced in docs (`scenario-pack`, `heroClassId`) but have no
  schemas.
- `AssetIndex` is listed but not in the schema-matrix dependency view.

Add schemas or explicitly defer them with a "planned, M3+" marker so
agents stop guessing.

### I2. Auto-balancer target (45–55% in 200 battles) is statistically optimistic
200 binomial trials at p=0.5 have a 95% CI of ±~7pp, so even a
perfectly balanced faction will flunk the 45–55% gate ~15% of the
time. Loosen to 40–60% or run 1,000 battles, and separate "balanced"
from "obviously broken" with a Wilson interval.

### I3. `M6` exit criteria is unreachable as written
"A user types '...' and receives a playable, balanced faction mod pack
within 3 minutes." 200 sim battles + image-gen + 2 LLM calls in 180 s
is aggressive. Either separate "coherent draft" (≤60 s) from "balanced"
(async, minutes–hours) or remove the clock.

### I4. No glossary / domain dictionary
AI agents have to reverse-engineer "stack", "dwelling", "paper doll",
"mastery tier", "neutral stack", "adventure map", "specialty" from
scattered prose. Add `docs/architecture/glossary.md` (one paragraph per
term, linked from `AGENTS.md`).

### I5. Task-file emoji taxonomy is undocumented
Every task has `🤖`, `🧠`, `⚠️`, sometimes combined. No file defines
the legend. Add a 5-line section to `tasks/README.md` explaining the
marks — otherwise an AI agent will make one up.

### I6. `research/deep-research-report.md` is a stub but is a named dependency
`tasks/mvp/09-tactical-combat/03-damage-formula-legacy-accurate.md`
references "legacy expansion-accurate" values that only exist here. The file is
currently a 13-line placeholder. Either populate the required tables
(unit stats, ATK/DEF caps, luck cap) or mark the downstream tasks
`BLOCKED: research/unit-stats-table.md`.

### I7. `capabilities` field has no enum
Manifest allows any capability string. With `sandboxed: true`
enforcement planned later, this should be a closed enum (e.g.
`["spells.custom-dsl", "scripts.js", "assets.binary"]`). Otherwise an
AI pack can claim any capability and the loader has to hardcode
policy.

### I8. Missing `scripts/validate-examples.mjs`
`scripts/check-repo-contracts.mjs` only validates `manifest.json`
examples against the manifest schema. It does not validate
`*.unit.json`, `*.hero.json`, `*.ability.json`, etc., against their
schemas. Trivial to add; critical to keep examples in sync as schemas
evolve.

### I9. Schema `$schema` paths in examples are relative and fragile
Examples use paths like
`"../../../../../content-schema/schemas/unit.schema.json"`. Moving one
folder silently breaks all of these. Use `$id` URIs
(`"heroes-reforged/unit.schema.json"`) resolved by the runtime instead
of filesystem paths.

## 5. Specific Fixes

### Renames / moves
1. Pick ONE: rename `tasks/mvp/04-faction-castle*` →
   `04-faction-emberwild*`, OR rename example pack
   `content-schema/examples/packs/emberwild-*` → `castle-*`.
2. `docs/planning/executable-backlog.md` and `tasks/README.md` overlap
   ~80%. Keep `tasks/README.md` as the execution board; make
   `executable-backlog.md` a 10-line pointer file or delete it.
3. Move `06b-visual-fidelity` into phase-2, or merge into
   `06-renderer`.

### Schema changes (concrete)
4. `unit.schema.json`: change `stats.*` from `"type": "number"` to
   `"type": "integer", "minimum": 0`. Same for `cost.*`, `growth.*`.
5. `ruleset.schema.json`: drop `formulas` as string map; add a
   referenced `formula.schema.json` with a structured op AST
   (`{"op": "mul", "args": [...]}`), or schedule a formal DSL task
   before M3.
6. `spell.schema.json`: lock root `additionalProperties: false`; add
   `"targeting": {"$ref": "targeting.schema.json"}`; introduce
   `"effects": [...]` with a discriminated union on `type` (`"damage"`,
   `"status"`, `"summon"`, `"modifier"`, `"dispel"`).
7. `ability.schema.json`: close effects via `oneOf`/discriminator on
   `type`; create a canonical list under
   `docs/architecture/effect-registry.md`.
8. `artifact.schema.json`: close root and `presentation`; structure
   `effects` like spells.
9. `manifest.schema.json`: add
   `contentHash: {type: "string", pattern: "^[a-f0-9]{64}$"}`
   optional-then-required, add `engine.hash`, close `capabilities` to
   an enum in a separate PR.
10. Add `skill.schema.json`, `hero-class.schema.json`,
    `scenario.schema.json`, `generation-request.schema.json`,
    `generated-faction.schema.json`.

### Task splits / additions
11. Add `tasks/mvp/02-content-schemas/12-formula-dsl.md` (~4h) —
    define DSL grammar, evaluator, test vectors.
12. Add `tasks/mvp/02-content-schemas/13-effect-registry.md` (~3h) —
    enumerate effect `type` values, subschemas.
13. Split `phase-3/02-ai-generation/03-auto-balancer` into (a)
    headless battle runner (~4h), (b) win-rate statistics + CI (~2h).
14. Add `phase-3/02-ai-generation/00-generation-io-schemas.md` (~2h) —
    `generation-request.schema.json` +
    `generated-faction.schema.json` BEFORE task 01.
15. Add `tasks/mvp/01-engine-core/07b-canonical-json.md` — canonical
    serializer is called out in overview but implicit in task 07
    ("serializer + hash"). Split for review clarity.

### Formula / balance fixes
16. Restate the legacy expansion damage formula using integer-only math:
    `numerator/denominator` constants stored in the ruleset; multiply
    first, divide last, with overflow guards. Add explicit test
    vectors for ATK−DEF = 0/1/10/28/29 and DEF−ATK = 0/1/10/28/29.
17. Add a luck-fires acceptance test to
    `03-damage-formula-legacy-accurate.md` (currently absent).
18. Loosen auto-balance target to 40–60% or use a 1,000-battle budget
    with Wilson CI.

### CI / tooling
19. Extend `scripts/check-repo-contracts.mjs` to validate every
    `*.{unit,hero,spell,artifact,building,ability}.json` example
    against its schema.
20. Add `scripts/check-id-namespaces.mjs` — enforce
    `<packId>:<kind>:<local>` ID pattern across examples.
21. Gitignore or minify `tasks/task-registry.json`.

## 6. AI-Friendliness Score

7.0 / 10.

### Where it succeeds
- `AGENTS.md` is explicit, short, and protects real invariants.
- `README.md` has an accurate "what exists / what doesn't" section.
- Task files have stable headers, explicit outputs, dependencies, and
  acceptance criteria.
- `tasks/task-registry.json` gives a machine-readable view.
- Clear "read first" canonical ordering.

### Where it will frustrate agents
- Conflicting faction names (C1) — the model will pick the wrong one
  ~30% of the time.
- Effect/spell/artifact schemas too permissive (C4) — generated JSON
  validates but is meaningless.
- Formula-as-string with no evaluator (C3) — agents will invent
  incompatible mini-DSLs per task.
- Emoji task taxonomy without legend — agents will guess.
- Missing glossary — agents re-derive terminology every session.
- Scattered "legacy expansion-accurate" references with no reference data checked
  in — tasks under-specified.
- `research/deep-research-report.md` is a stub but several tasks
  transitively depend on it.

### One-line AI verdict
An AI agent can pick a task and produce something; it cannot reliably
produce something that composes correctly with the rest of the system,
because the schemas don't pin the hard parts.

## 7. Top Risks

1. IP risk: "legacy expansion-accurate" peppered through combat tasks (legacy expansion, classic fantasy strategy). Renaming formulas to "classic turn-based" and
   calibrating via published fan-reverse-engineering rather than
   asserting parity to a specific commercial product is safer.
2. Desync risk: Without `contentHash` in manifest (C5) and with floats
   leaking through stats (C2), two clients with the same `version`
   string can deterministically disagree.
3. Sandbox risk: Formulas-as-strings + planned community packs + no
   DSL = direct path to code injection in the browser.
4. Scope risk: M1 "strategic vertical slice" currently depends on
   tasks across 5 MVP modules that don't exist as code — the M0→M1
   ramp is steeper than the roadmap implies.
5. Contract drift: Schemas and docs already disagree in places (C1,
   C2, pack-contract vs example). Without schema-driven example
   validation (I8), this gets worse with every commit.
6. AI-generation UX risk: 3-minute balanced-faction SLA (I3) will
   embarrass the project at demo time. Redefine the promise.

## 8. Suggested Order of Operations

1. Resolve C1 (Castle/Emberwild).
2. Tighten C2 (integer stats) + C4 (effect registry) — pure schema
   PRs; zero code needed.
3. Write the glossary (I4) and emoji legend (I5) — 15 minutes of
   value-to-effort.
4. Add `contentHash` to manifest (C5) — one-line schema change;
   unblocks future MP/replay work.
5. Specify the formula DSL (C3) or switch to AST-JSON.
6. Extend the validator to cover all example records (I8, fix #19).
7. Add generation I/O schemas (C6).

Everything else can wait until real code starts landing in `src/`.

---

## Bottom Line

The repo is a B+ plan with A+ packaging. The packaging is so good it
hides the fact that the three most important contracts in the project
(effects, formulas, generation I/O) are barely specified. Fix those
before a single line of engine code gets written — retrofitting
schemas after a working engine exists is where projects like this go
to die.
