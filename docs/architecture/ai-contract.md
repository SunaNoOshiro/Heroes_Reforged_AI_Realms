# AI Runtime Contract

Single source of truth for the gameplay-opponent AI runtime
(heuristic + MCTS). All AI task files cite this doc for the
contract; task files keep their implementation notes but do not
re-state the contract here.

This doc covers gameplay AI only. Provider-backed content-generation
AI is governed separately by
[`ai-integration.md`](./ai-integration.md) and
[`ai-generation-pipeline.md`](./ai-generation-pipeline.md).

## 1. Input View — `aiPlayerView(state, playerId, cheats)`

The worker never receives raw `AdventureState`. The dispatcher
projects per-player state through `aiPlayerView` before crossing
the worker boundary.

```ts
type AiCheats = {
  seeFog?: boolean;        // bypass fog-of-war stripping
  resourceBonus?: number;  // additive multiplier on daily income
};

aiPlayerView(state: AdventureState, playerId: PlayerId, cheats?: AiCheats): AdventureView
```

**Default** for every difficulty in M2 / M3 / M7 is `cheats = {}`.
Higher cheat tiers are an explicit opt-in, not a side-channel.

**Stripped fields (when `cheats.seeFog !== true`):**

- Tiles outside the player's current visibility map.
- Opponent hero `inventory`, `spellBook`, `path`, and any
  unrevealed equipment.
- Map objects on hidden tiles.
- Opponent town garrison composition, build queue, and gold
  reserves.
- Any `AdventureState` field marked `private: true` per
  [`state-shape.md`](./state-shape.md).

**Retained fields:**

- Self-owned heroes, towns, resources in full.
- Visible map objects and visible opponent hero positions.
- Public scenario metadata (turn, day, weather, scenario id).

The `SCOUT_FOG` want type
([`tasks/mvp/10-heuristic-ai/02-wants-engine-strategic-action-prioritization.md`](../../tasks/mvp/10-heuristic-ai/02-wants-engine-strategic-action-prioritization.md))
operates over the projected view: the AI scouts what it cannot see.

The dispatcher validates emitted commands against full
`AdventureState`. The view is the AI's perception; the dispatcher
is the source of truth for legality.

Implementing task:
[`tasks/mvp/10-heuristic-ai/07-ai-player-view-projection.md`](../../tasks/mvp/10-heuristic-ai/07-ai-player-view-projection.md).

## 2. Output — `Command`

The AI emits the same closed `Command` enum as a human player.
There are no AI-only command kinds. See
[`command-schema.md`](./command-schema.md) for the full alphabet.

The worker returns one `Command` per `requestAIMove` call. Multi-
step turns are represented as a sequence of `requestAIMove` calls,
one per dispatched command, until the AI returns `END_HERO_TURN`
or `END_DAY`.

Stable invariants:

- The returned `Command` carries a deterministic `metadata.nonce`
  generated by the AI actor. See
  [`command-schema.md` § Deduplication](./command-schema.md#deduplication).
- The AI never invents entity IDs; it consumes the deterministic
  allocator per [`id-allocator.md`](./id-allocator.md).
- A returned `Command` may be rejected by the dispatcher if it is
  illegal against full state. The dispatcher's verdict is final;
  the AI never retries on its own.

## 3. Worker Protocol

```
type RequestId = string;

// dispatcher → worker
{ type: "COMPUTE_MOVE",
  requestId: RequestId,
  view: AdventureView,
  difficulty: DifficultyLevel,
  searchBudget: { maxNodes: number, maxDepth: number },
  cheats?: AiCheats,
  rngSeed: string }

// worker → dispatcher
{ type: "MOVE_RESULT",
  requestId: RequestId,
  command: Command,
  nodesExpanded: number,
  searchDepthReached: number }

// dispatcher → worker (opt-in dev-only; see § 7)
{ type: "AI_TRACE_REQUEST",
  requestId: RequestId,
  view: AdventureView,
  difficulty: DifficultyLevel,
  rngSeed: string }

// worker → dispatcher (opt-in dev-only)
{ type: "AI_TRACE_RESULT",
  requestId: RequestId,
  view: AdventureView,
  wants: Want[],
  scored: ScoredAction[],
  command: Command,
  reasoning: string }
```

Lifecycle:

- Spawn: one worker per active AI player. Workers may be pre-warmed
  across turns (see § 6).
- Termination: `Worker.terminate()` is reserved for "game ends" /
  "user navigates away". It is **not** the cancellation path. See
  § 5.
- Error path: the worker reports `{ type: "AI_ERROR", requestId,
  reason }` and the dispatcher falls back to `END_HERO_TURN` /
  `END_DAY` per § 4.

The closed envelope schema is
[`content-schema/schemas/worker-message.schema.json`](../../content-schema/schemas/worker-message.schema.json).
Every `kind` value is reflected in the `kind` enum and the matching
`payload` `oneOf` branch:

| `kind`              | Direction       | Owner task                                                                                                       | Notes                                              |
|---------------------|-----------------|------------------------------------------------------------------------------------------------------------------|----------------------------------------------------|
| `COMPUTE_MOVE`      | main → worker   | [`tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md`](../../tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md) | Carries `stateView` + `difficulty`.                |
| `MOVE_RESULT`       | worker → main   | [`tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md`](../../tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md) | Carries the chosen `Command` + optional log.       |
| `ABORT`             | main → worker   | [`tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md`](../../tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md) | Cancellation gate per § 5.                         |

## 4. Per-Turn Budget Table

Per-call budgets remain in their respective tasks (50 ms threat
map, 5 ms per `evaluateActions`). The whole-turn aggregate cap
below replaces the previous single 2 s hard timeout.

| Difficulty   | `maxNodes`     | `maxDepth`     | `wallClockHardMs` | Per-turn budget (soft) | No-action fallback |
|--------------|----------------|----------------|-------------------|------------------------|--------------------|
| Pawn         | 4 000          | 3              | 500               | 200 ms                 | `END_HERO_TURN`    |
| Knight       | 16 000         | 5              | 1 000             | 500 ms                 | `END_HERO_TURN`    |
| Grand Master | (set by M3)    | (set by M3)    | 2 000             | 1 s                    | `END_HERO_TURN`    |
| Lord         | (set by M7)    | (set by M7)    | 4 000             | 2 s                    | best-of-MCTS-rollouts so far, else `END_HERO_TURN` |
| Immortal     | (set by M7)    | (set by M7)    | 6 000             | 3 s                    | best-of-MCTS-rollouts so far, else `END_HERO_TURN` |

This table is the **only authoritative source** for AI search-budget
constants. Implementing tasks (heuristic, MCTS, performance docs)
import the values from here; do not re-state them elsewhere. The
`(set by Mn)` entries are filled in by the listed implementing task
when the matching milestone lands.

Rules:

- The "Per-turn budget" is the soft cap. Search continues up to
  this budget; on overflow the worker returns the best-found
  command immediately on completion of the current
  `evaluateActions` call (≤ 5 ms).
- The "Hard timeout" is the cap that fires the cancellation path
  (§ 5). On hard-timeout fire, the worker MUST return a valid
  `Command` — best-found if any has scored, otherwise the
  per-difficulty no-action fallback.
- The dispatcher always sees a valid `Command`; there is no
  "AI returned nothing" path.
- Cross-machine search-budget determinism (the existing
  `determinism.md` AI Compute Budget guarantee) holds for the
  search itself: identical seed + state + budget on two runs
  produces an identical `Command` *when the search finishes within
  budget*. If the hard timeout fires, the no-action fallback is
  used; the resulting `Command` is logged once and replays
  bit-identically from the log without re-running the search.

Implementing tasks:

- Pawn / Knight rows: [`tasks/mvp/10-heuristic-ai/05-difficulty-levels-pawn-and-knight.md`](../../tasks/mvp/10-heuristic-ai/05-difficulty-levels-pawn-and-knight.md)
- Grand Master row: [`tasks/phase-2/02-strategic-ai/05-grand-master-difficulty-plus-headless-evaluation.md`](../../tasks/phase-2/02-strategic-ai/05-grand-master-difficulty-plus-headless-evaluation.md)

## 5. Cancellation

```
requestAIMove(view, difficulty, signal?: AbortSignal): Promise<Command>
```

When `signal` aborts (caller-driven, save-system-driven, or
hard-timeout-driven per § 4), the worker MUST:

1. Finish the current `evaluateActions` call (≤ 5 ms by existing
   per-call budget).
2. Return the best-found `Command` if any action has scored; else
   return the per-difficulty no-action fallback from § 4.
3. Resolve (not reject) the returned `Promise` with that
   `Command`. The dispatcher always sees a valid `Command`.

`Worker.terminate()` is reserved for game-end / navigate-away. It
is not the cancellation path, because terminate-then-spawn loses
warm caches and pre-computed pathfinder state.

Required tests (in [`tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md`](../../tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md)):

- "abort during `buildThreatMap` returns `END_HERO_TURN`
  deterministically".
- "abort after one `evaluateActions` call returns the scored
  best-action deterministically".

## 6. Parallelism

AI players act **sequentially in turn order**. Workers may be
pre-warmed (kept resident across turns) but commands always apply
to the dispatcher in turn order. Multi-worker parallel compute is
permitted only as an internal optimization that does not affect
the order of `Command`s appended to the log.

This rule complements
[`command-schema.md` § Cross-Actor Ordering](./command-schema.md#cross-actor-ordering)
(AI co-actors are treated as players with deterministic
`playerId`s; same rule as hotseat).

Multi-worker parallel pre-warming is policy-only in MVP — no
optimization is implemented. Promotion is opt-in for M7 if
profiling demands it.

### AI Determinism Under Wall-Clock Budgets

Wall-clock hard timeouts (`wallClockHardMs` per § 4) introduce a
non-deterministic decision point: machine speed determines whether
the timeout fires before search converges. The replay invariant
holds anyway because the chosen `Command` is logged and replays
read from the log — but only if exactly **one** AI search produces
the command for a given (turn, player) pair. This section pins
the structural rule that keeps that invariant true across all
play modes:

- **Wall-clock budgets are permitted only in the broadcaster-elected
  AI worker** per
  [`tasks/phase-3/01-multiplayer/07-host-migration-heartbeat-election.md`](../../tasks/phase-3/01-multiplayer/07-host-migration-heartbeat-election.md).
  The broadcaster runs the search; the chosen `Command` is logged
  to the lockstep stream.
- **Non-broadcaster peers MUST NOT re-run AI search.** They consume
  the broadcast `Command` from the lockstep log. Independently
  re-running search on each peer would diverge the moment any
  peer's wall-clock fires before another's.
- **Single-player and friendly-MP-bot paths route through the same
  broadcaster gate**: the local player is the "broadcaster" by
  definition in single-player; friendly MP elects a broadcaster
  per the heartbeat-election task above.
- The rule is enforced by the existing `botRngStreamId` model
  pinned by Bot RNG Sub-Streams: only the elected
  broadcaster mints `botRngStreamId` consumers; other peers read
  the resulting `Command` from the log without minting their own
  RNG draws.

Future N-peer mesh designs (deferred per `glossary.md` M7
sketches) MUST preserve this rule: if a feature lets each peer
run its own AI search under a wall-clock budget, the feature is
incompatible with deterministic replay and is rejected at design
review.

## 7. Decision Log

The canonical command log is bit-identical replay's source of
truth and MUST NOT carry AI internals. The opt-in `aiDecisionLog`
side channel records AI traces without affecting the replay hash.

```ts
type AdventureViewSnapshotHash = string; // xxh64 hex of canonical view

type AiDecisionLogEntry = {
  turn: number;
  playerId: PlayerId;
  difficulty: DifficultyLevel;
  view: AdventureViewSnapshotHash;
  wants: Want[];
  scored: ScoredAction[];
  chosen: Command;
  reasoning: string;
};
```

Rules:

- Enabled by `Engine.config.aiDecisionLog: boolean`. Default
  `false`. When `false`, no allocation occurs.
- When `true`, entries are written to an in-memory ring buffer of
  the last N turns (N pinned by the implementing task) and may be
  serialized to disk for QA.
- The replay hash and saves are unchanged whether the flag is
  on or off. CI asserts hash parity.
- The dev-only AI inspector screen (§ below) is the UI consumer
  of `aiDecisionLog` plus the `AI_TRACE_*` worker messages.

Implementing task:
[`tasks/mvp/10-heuristic-ai/09-ai-decision-log-channel.md`](../../tasks/mvp/10-heuristic-ai/09-ai-decision-log-channel.md).

UI surface: dev-only screen package
[`docs/architecture/wiki/screens/69-dev-ai-inspector/`](./wiki/screens/69-dev-ai-inspector/),
implemented by
[`tasks/mvp/10-heuristic-ai/08-ai-inspector-dev-screen.md`](../../tasks/mvp/10-heuristic-ai/08-ai-inspector-dev-screen.md).

## 8. BotProvider

Tests, the bench harness, and future MCTS regression all swap the
opponent through one interface:

```ts
type BotProvider = {
  id: string;
  requestAIMove(
    view: AdventureView,
    difficulty: DifficultyLevel,
    signal?: AbortSignal
  ): Promise<Command>;
};
```

First-party implementations:

- `heuristicBot` — M2 default; wraps the worker.
- `randomBot(seed)` — picks a uniformly-random legal action; the
  baseline opponent for the Knight quality gate.
- `scriptedBot(commands)` — replays a fixed `Command[]` for smoke
  tests.

Tests pick a provider by `id`. Production wires `heuristicBot`
only. M7 adds `mctsBot` as a fourth implementation without
changing call sites.

Implementing task:
[`tasks/mvp/10-heuristic-ai/10-bot-provider-interface.md`](../../tasks/mvp/10-heuristic-ai/10-bot-provider-interface.md).

## 9. Cheats

Cheats are explicit opt-in flags carried on the worker request.

```ts
type AiCheats = {
  seeFog?: boolean;        // bypass § 1 fog-of-war stripping
  resourceBonus?: number;  // additive multiplier on daily income
};
```

Defaults for shipped difficulties:

| Difficulty   | Default cheats | Notes |
|--------------|----------------|-------|
| Pawn         | `{}`           | Fair AI. |
| Knight       | `{}`           | Fair AI. Required for the 80 % vs random quality gate to be meaningful. |
| Grand Master | `{}`           | Fair AI. |
| Lord         | `{}`           | Fair AI. |
| Immortal     | `{}`           | Fair AI by default; cheat slots reserved for community / scenario use. |

Higher cheat tiers (e.g. scenario-defined "AI sees fog") are
content-driven and pinned per scenario or pack. Engine code never
defaults a cheat to `true`.
