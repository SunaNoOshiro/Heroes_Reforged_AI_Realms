## 1. Input View â€” `aiPlayerView(state, playerId, cheats)`

The worker never receives raw `AdventureState`. The dispatcher
projects per-player state through `aiPlayerView` before crossing the
worker boundary.

```ts
type AiCheats = {
  seeFog?: boolean;        // bypass fog-of-war stripping
  resourceBonus?: number;  // additive multiplier on daily income
};

aiPlayerView(state: AdventureState, playerId: PlayerId, cheats?: AiCheats): AdventureView
```

Default for every difficulty in M2 / M3 / M7 is `cheats = {}`. Higher
cheat tiers are an explicit opt-in, not a side channel.

Stripped when `cheats.seeFog !== true`:

- Tiles outside the player's current visibility map.
- Opponent hero `inventory`, `spellBook`, `path`, and any unrevealed
  equipment.
- Map objects on hidden tiles.
- Opponent town garrison composition, build queue, and gold reserves.
- Any `AdventureState` field marked `private: true` per
  [`state-shape.md`](./state-shape.md).

Retained:

- Self-owned heroes, towns, resources in full.
- Visible map objects and visible opponent hero positions.
- Public scenario metadata (turn, day, weather, scenario id).

The `SCOUT_FOG` want type
([`tasks/mvp/10-heuristic-ai/02-wants-engine-strategic-action-prioritization.md`](../../tasks/mvp/10-heuristic-ai/02-wants-engine-strategic-action-prioritization.md))
operates over the projected view: the AI scouts what it cannot see.

The dispatcher validates emitted commands against full
`AdventureState`. The view is the AI's perception; the dispatcher is
the source of truth for legality.

Implementing task:
[`tasks/mvp/10-heuristic-ai/07-ai-player-view-projection.md`](../../tasks/mvp/10-heuristic-ai/07-ai-player-view-projection.md).

## 2. Output â€” `Command`

The AI emits the same closed `Command` enum as a human player. There
are no AI-only kinds. See
[`command-schema.md`](./command-schema.md) for the full alphabet.

The worker returns one `Command` per `requestAIMove` call. Multi-step
turns are a sequence of `requestAIMove` calls, one per dispatched
command, until the AI returns `END_HERO_TURN` or `END_DAY`.

Stable invariants:

- The returned `Command` carries a deterministic `metadata.nonce`
  generated by the AI actor â€” see
  [`command-schema.md` Â§ Deduplication](./command-schema.md#deduplication).
- The AI never invents entity IDs; it consumes the deterministic
  allocator per [`id-allocator.md`](./id-allocator.md).
- A returned `Command` may be rejected by the dispatcher if it is
  illegal against full state. The dispatcher's verdict is final; the
  AI never retries on its own.

## 4. Per-Turn Budget Table

Per-call budgets remain in their respective tasks (50 ms threat map,
5 ms per `evaluateActions`). The whole-turn aggregate cap below
replaces the previous single 2 s hard timeout.

| Difficulty   | `maxNodes`     | `maxDepth`     | `wallClockHardMs` | Per-turn budget (soft) | No-action fallback |
|--------------|----------------|----------------|-------------------|------------------------|--------------------|
| Pawn         | 4 000          | 3              | 500               | 200 ms                 | `END_HERO_TURN`    |
| Knight       | 16 000         | 5              | 1 000             | 500 ms                 | `END_HERO_TURN`    |
| Grand Master | (set by M3)    | (set by M3)    | 2 000             | 1 s                    | `END_HERO_TURN`    |
| Lord         | (set by M7)    | (set by M7)    | 4 000             | 2 s                    | best-of-MCTS-rollouts so far, else `END_HERO_TURN` |
| Immortal     | (set by M7)    | (set by M7)    | 6 000             | 3 s                    | best-of-MCTS-rollouts so far, else `END_HERO_TURN` |

This table is the **only authoritative source** for AI search-budget
constants. Implementing tasks (heuristic, MCTS, performance docs)
import these values; do not re-state them elsewhere. The
`(set by Mn)` entries are filled in by the listed implementing task
when the matching milestone lands.

Rules:

- The "Per-turn budget" is the soft cap. Search continues up to this
  budget; on overflow the worker returns the best-found command
  immediately on completion of the current `evaluateActions` call
  (â‰¤ 5 ms).
- The "Hard timeout" fires the cancellation path (Â§ 5). On
  hard-timeout fire, the worker MUST return a valid `Command` â€”
  best-found if any has scored, otherwise the per-difficulty
  no-action fallback.
- The dispatcher always sees a valid `Command`; there is no "AI
  returned nothing" path.
- Cross-machine search-budget determinism (the
  [`determinism.md` AI Compute Budget guarantee](./determinism.md#ai-compute-budget))
  holds for the search itself: identical seed + state + budget on two
  runs produces an identical `Command` *when the search finishes
  within budget*. If the hard timeout fires, the no-action fallback
  is used; the resulting `Command` is logged once and replays
  bit-identically from the log without re-running the search.

## 5. Cancellation

```
requestAIMove(view, difficulty, signal?: AbortSignal): Promise<Command>
```

When `signal` aborts (caller-driven, save-system-driven, or
hard-timeout-driven per Â§ 4), the worker MUST:

1. Finish the current `evaluateActions` call (â‰¤ 5 ms by existing
   per-call budget).
2. Return the best-found `Command` if any action has scored; else
   return the per-difficulty no-action fallback from Â§ 4.
3. Resolve (not reject) the returned `Promise` with that `Command`.
   The dispatcher always sees a valid `Command`.

`Worker.terminate()` is reserved for game-end / navigate-away. It is
not the cancellation path, because terminate-then-spawn loses warm
caches and pre-computed pathfinder state.

Required tests (in
[`tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md`](../../tasks/mvp/10-heuristic-ai/06-run-ai-in-web-worker.md)):

- "abort during `buildThreatMap` returns `END_HERO_TURN`
  deterministically".
- "abort after one `evaluateActions` call returns the scored
  best-action deterministically".

## 6. Parallelism

AI players act **sequentially in turn order**. Workers may be
pre-warmed (kept resident across turns) but commands always apply to
the dispatcher in turn order. Multi-worker parallel compute is
permitted only as an internal optimization that does not affect the
order of `Command`s appended to the log.

This rule complements
[`command-schema.md` Â§ Cross-Actor Ordering](./command-schema.md#cross-actor-ordering)
(AI co-actors are treated as players with deterministic `playerId`s;
same rule as hotseat).

Multi-worker parallel pre-warming is policy-only in MVP â€” no
optimization is implemented. Promotion is opt-in for M7 if profiling
demands it.

### AI Determinism Under Wall-Clock Budgets

Wall-clock hard timeouts (`wallClockHardMs` per Â§ 4) introduce a
non-deterministic decision point: machine speed determines whether
the timeout fires before search converges. The replay invariant holds
anyway because the chosen `Command` is logged and replays read from
the log â€” but only if exactly **one** AI search produces the command
for a given (turn, player) pair. This section pins the structural
rule that keeps that invariant true across all play modes:

- **Wall-clock budgets are permitted only in the broadcaster-elected
  AI worker** per
  [`tasks/phase-3/01-multiplayer/07-host-migration-heartbeat-election.md`](../../tasks/phase-3/01-multiplayer/07-host-migration-heartbeat-election.md).
  The broadcaster runs the search; the chosen `Command` is logged to
  the lockstep stream.
- **Non-broadcaster peers MUST NOT re-run AI search.** They consume
  the broadcast `Command` from the lockstep log. Independently
  re-running search on each peer would diverge the moment any peer's
  wall-clock fires before another's.
- **Single-player and friendly-MP-bot paths route through the same
  broadcaster gate**: the local player is the "broadcaster" by
  definition in single-player; friendly MP elects a broadcaster per
  the heartbeat-election task above.
- The rule is enforced by the existing `botRngStreamId` model pinned
  in [`determinism.md` Â§ Bot RNG Sub-Streams](./determinism.md#bot-rng-sub-streams):
  only the elected broadcaster mints `botRngStreamId` consumers;
  other peers read the resulting `Command` from the log without
  minting their own RNG draws.

Future N-peer mesh designs (deferred per
[`glossary.md` Â§ M7 Multiplayer Scope Sketches](./glossary.md#m7-multiplayer-scope-sketches))
MUST preserve this rule: if a feature lets each peer run its own AI
search under a wall-clock budget, the feature is incompatible with
deterministic replay and is rejected at design review.

## 7. Decision Log

The canonical command log is bit-identical replay's source of truth
and MUST NOT carry AI internals. The opt-in `aiDecisionLog` side
channel records AI traces without affecting the replay hash.

```ts
type AdventureViewSnapshotHash = string; // xxh64 hex of canonical view

type AiDecisionLogEntry = {
  turn: number;
  playerId: PlayerId;
  difficulty: DifficultyLevel;
  view: AdventureViewSnapshotHash;
  wants: Want[];
  scored: ScoredAction[];
  chosen: Command;
  reasoning: string;
};
```

Rules:

- Enabled by `Engine.config.aiDecisionLog: boolean`. Default `false`.
  When `false`, no allocation occurs.
- When `true`, entries are written to an in-memory ring buffer of the
  last N turns (N pinned by the implementing task) and may be
  serialized to disk for QA.
- The replay hash and saves are unchanged whether the flag is on or
  off. CI asserts hash parity.
- The dev-only AI inspector screen (UI surface below) is the consumer
  of `aiDecisionLog` plus the `AI_TRACE_*` worker messages.

Implementing task:
[`tasks/mvp/10-heuristic-ai/09-ai-decision-log-channel.md`](../../tasks/mvp/10-heuristic-ai/09-ai-decision-log-channel.md).

UI surface: dev-only screen package
