# 8. PERSISTENCE / SAVE SYSTEM

### Q: 150. Are saves full state, delta from base, or command log + seed?

**Status:** ⚠ Partial

**Answer:**
**Command-log + seed**, per the canonical Persistence task. A `SaveRecord` carries metadata, the seed, the ruleset id, the loaded content-pack hashes, the turn number, the **full command log**, and an optional `Checkpoint[]`. Loading replays the log from `seed` to reconstruct state — there is *no* live state in the spec.

There is a documentation inconsistency: the save-flow diagram (`24-save-flow.md`) shows a save file containing a `state` blob alongside `commandLog`. The MVP task `02-log-only-save-format.md` is explicit that "a save file contains only the metadata and command log — no game state." The implementation contract in the task list takes precedence; the diagram appears to be stale shorthand.

**Evidence:**
- `tasks/mvp/08-persistence/02-log-only-save-format.md` (`SaveRecord` shape; "no game state")
- `docs/architecture/diagrams/24-save-flow.md` (shows a `state` field — conflicts with above)
- `docs/architecture/diagrams/25-load-flow.md` ("Save initial scenario state (small) … Save command log (compact, deterministic) … On load: replay commands to reach current state")
- `docs/architecture/determinism.md` (Replay API: seed + command log reproduces final state)

### Q: 151. Are saves byte-identical given identical state?

**Status:** ⚠ Partial

**Answer:**
**For the canonical payload, yes.** The serializer that produces the save's logical content is the engine's canonical JSON serializer (sorted keys, no `undefined`/`NaN`/`Infinity`, fixed-point integers — see `01-engine-core/07-state-serializer-plus-xxh64-hash`), so a given `(seed, commandLog, ruleset, packHashes, turnNumber)` produces a byte-identical canonical pre-compression buffer.

**On-disk bytes are not guaranteed identical**, however, because:
- Saves are gzip-compressed, and compression-level / library determinism is not pinned (see Q154).
- The `metadata` block holds wall-clock `createdAt` / `savedAt` timestamps and a `name` chosen by the player, which differ between saves of the same logical state.
- An autogenerated `id` (UUID) is assigned per save record.

There is no "save A == save B byte-for-byte if same state" acceptance test in the spec.

**Evidence:**
- `tasks/mvp/01-engine-core/07-state-serializer-plus-xxh64-hash.md`
- `docs/architecture/determinism.md` ("Canonical serializer + state hash (sorted keys, no whitespace, xxh64 over canonical bytes)")
- `tasks/mvp/08-persistence/02-log-only-save-format.md` (`createdAt`, `savedAt`, `id`, `name` fields → not state-derived)
- `docs/architecture/diagrams/24-save-flow.md` (Compress gzip step, level unspecified)

### Q: 152. Is replay = save format, or distinct?

**Status:** ⚠ Partial

**Answer:**
**Effectively the same.** The save *is* the replay artifact: the command log + seed + content/engine hashes are exactly what `replay(seed, log)` consumes. Loading a save **is** a replay (verified by re-computing the state hash and comparing against the one recorded at save time). There is no separate replay file format documented.

What is **not** specified is whether replays carry a different envelope when shared (e.g., for spectator playback, bug reports, or determinism harness fixtures). The fuzz harness writes its own fixtures (`docs/architecture/determinism.md`), but the wire format is not declared.

**Evidence:**
- `docs/architecture/diagrams/25-load-flow.md` (load == replay)
- `tasks/mvp/01-engine-core/08-replay-api.md` (Replay API consumes seed + command log)
- `docs/architecture/determinism.md` ("Replay API (seed + command log reproduces final state)")
- No standalone `replay-format.md` or `*.replay` schema in the repo.

### Q: 153. Can a replay be reproduced bit-exact on a different machine?

**Status:** ✔ Defined

**Answer:**
**Yes — this is the central determinism contract.** Identical `seed` + `commandLog` + `contentPackHashes` + engine version produce a bit-identical state on any machine, any OS, Node or browser. Determinism is enforced by:

- Seeded RNG (PCG32 with named sub-streams).
- Fixed-point integer math (no JS floats in gameplay).
- Canonical JSON serialization (sorted keys, normalized).
- xxh64 state hash recomputed and verified after replay.
- Forbidden APIs in deterministic paths: `Math.random`, `Date.now`, `performance.now`, `eval`, async-timing decisions, unsorted Map/Set iteration.
- Fuzz harness (M1 task 9): 1000-command AI-vs-AI determinism test, must replay bit-identical.

The load flow explicitly verifies this: post-replay state hash mismatch ⇒ "Save corrupt!" error.

**Evidence:**
- `docs/architecture/determinism.md` ("Hard constraint: same seed + same commands + same content hashes ⇒ same state on any machine")
- `tasks/mvp/01-engine-core/08-replay-api.md`
- `tasks/mvp/01-engine-core/07-state-serializer-plus-xxh64-hash.md`
- `docs/architecture/diagrams/25-load-flow.md` (post-replay hash check P)

### Q: 154. Are saves compressed, and is compression deterministic?

**Status:** ⚠ Partial

**Answer:**
**Compressed: yes (gzip).** **Deterministic: not specified.** The save flow ends with a "Compress gzip → Write to disk" step but no compression level, library (`pako`, `zlib`, `CompressionStream`), or determinism guarantee is documented. Most gzip implementations are deterministic at a fixed level given identical input, but this is an implicit assumption rather than a contract.

Save sizes documented:
- `02-log-only-save-format.md`: < 50 KB for a 7-day game.
- `24-save-flow.md`: typically 50–200 KB compressed.
The two figures disagree mildly — both are order-of-magnitude estimates rather than budgets.

**Evidence:**
- `docs/architecture/diagrams/24-save-flow.md` ("Compress gzip", "~50-200 KB")
- `tasks/mvp/08-persistence/02-log-only-save-format.md` ("< 50KB for a 7-day game")
- No file specifies gzip level, library, or byte-equivalence under recompression.

### Q: 155. Is save format versioned, with migration scripts?

**Status:** ⚠ Partial

**Answer:**
**Versioned: yes.** Each save carries a `saveVersion` integer (currently `1`). The load pipeline validates **schema version, content hashes, pack compatibility, ruleset version, and migration availability** before hydrating state. The Save/Load screen's `selectors.persistence.selectedSaveCompatibility` exposes the result as "Version/hash/migration result," and the missing-states catalog declares an "**incompatible save migration needed**" state.

**Migration scripts: not yet authored.** The spec defines the *gates* (validate version, surface "migration available?") and the UI (compatibility seal, "Schema migration: none" / migration result indicator), but no migration registry, transform-function format, or example migrator is provided. The IndexedDB wrapper auto-upgrades its own schema on version bump, but that governs the database layout, not save-record migrations.

### Q: 157. What happens on truncated, corrupted, or tampered saves?

**Status:** ✔ Defined

**Answer:**
**Fail-loud, three gates:**

1. **Format gate** — `Valid format?` after decompression. NO ⇒ "Show error" terminal node, do not load.
2. **Pack-hash gate** — `All packs match?`. NO ⇒ user is warned with "continue or abort?" choice.
3. **Replay-hash gate** — after replaying every command, `Hash matches?`. NO ⇒ "Save corrupt!" terminal node.

Truncated logs and tampered commands both surface as a final-hash mismatch (gate 3); a totally garbled file fails gate 1. UI policy: keep current screen open, preserve drafts where useful, show localized error text, play failure feedback. The data-contracts file states explicitly: "Missing gameplay records, invalid commands, and unresolved content IDs fail loudly before controls become enabled."

The MVP import path also enforces this: "Importing a corrupted file shows a clear error (not a crash)" (acceptance criterion).

**Evidence:**
- `docs/architecture/diagrams/25-load-flow.md` (gates D, G, P; terminal nodes E "Show error" and Q "Save corrupt!")
- `docs/architecture/wiki/screens/55-save-load/data-contracts.md` ("fail loudly")
- `docs/architecture/wiki/screens/55-save-load/interactions.md` ("On rejection, keep the current screen open, preserve local draft when useful, show localized error text, and play failure feedback.")
- `docs/architecture/wiki/missing-states.md` ("corrupted save")
- `tasks/mvp/08-persistence/05-export-import-json.md` (Acceptance: corrupted import fails cleanly)

### Q: 158. Is there a checksum or signature on save files?

**Status:** ⚠ Partial

**Answer:**
**Checksum: yes (xxh64).** **Cryptographic signature: no.**

Saves carry:
- `stateHash` — xxh64 over the canonical serialization of the post-replay `GameState`. Re-computed on load and compared, providing end-to-end tamper detection of the command log + seed + reducer.
- `contentPackHashes[]` — xxh64 per loaded pack, pinned at save time and re-validated on load.
- (Implicit) `engineHash` from the manifest schema, intended to pin engine build for reproducibility (post-M2 mandatory; pre-M2 optional).

Hashes are **non-keyed** (xxh64 is fast but not cryptographic). There is no HMAC, no public-key signature, and no signed-manifest authentication. A motivated user can re-author the command log and re-hash; the system detects accidental corruption and replay drift, not adversarial forgery.

**Evidence:**
- `tasks/mvp/01-engine-core/07-state-serializer-plus-xxh64-hash.md`
- `docs/architecture/determinism.md` ("xxh64 over canonical bytes")
- `docs/architecture/diagrams/24-save-flow.md` (`"stateHash": "x7y8z9..."`)
- `content-schema/schemas/manifest.schema.json` (`contentHash`, `engineHash`)
- No HMAC / signature scheme in the repo.

### Q: 159. Where are saves stored (filesystem, IndexedDB, cloud)?

**Status:** ✔ Defined

**Answer:**
- **Primary:** **IndexedDB**, via the wrapper at `src/persistence/db.ts`. Three object stores: `saves`, `scenarios`, `content`. Per-store interface: `get`/`set`/`delete`/`list`. Auto-upgrades schema on version bump. Targets Chrome / Firefox / Safari with no fallbacks for MVP.
- **Manual export/import:** browser File API. Saves serialize to `save-name.hrsa.json` files for backup to user-managed cloud (Drive / Dropbox).
- **No first-party cloud sync** for MVP. No server-side save store, no account system, no automatic backup.

**Evidence:**
- `tasks/mvp/08-persistence/01-indexeddb-wrapper.md`
- `tasks/mvp/08-persistence/05-export-import-json.md` (`save-name.hrsa.json`)
- No cloud-save task in `tasks/mvp/`, `tasks/phase-2/`, or `tasks/phase-3/`.

### Q: 161. Are saves atomic (write-temp-then-rename)?

**Status:** ❌ UNKNOWN

**Answer:**
**Not specified.** The save-flow diagram ends with "Write to disk" with no transactional contract.

In practice:
- **IndexedDB writes are transactional by design** at the API level — a `set(id, data)` either commits or rolls back; the file appears whole or not at all. So the "write-temp-then-rename" pattern is unnecessary for the primary path.
- **JSON export / import** uses the browser File API and does not have an equivalent atomicity guarantee. A crash mid-download would yield a truncated file on disk, which the load gates would reject (not silently corrupt the player's library).

What is **not** documented:
- Whether the IndexedDB wrapper opens a single transaction spanning all sub-writes (manifest + payload) or two separate ones (race window between).
- Whether autosave writes to a "shadow" slot first and only swaps after replay-verification, or overwrites in place.
- Whether export uses `URL.createObjectURL` ⇒ download (synchronous from the user's POV) or streams.
- Crash-safety semantics on a tab kill mid-save.

**Evidence:**
- `docs/architecture/diagrams/24-save-flow.md` ("Write to disk" — no atomicity contract)
- `tasks/mvp/08-persistence/01-indexeddb-wrapper.md` (no mention of transactions or atomic swap)
- `tasks/mvp/08-persistence/05-export-import-json.md` (no atomicity for File API path)

### Q: 162. Can a save be partially loaded for inspection without full deserialization?

**Status:** ✔ Defined

**Answer:**
**Yes — manifest-only inspection is the default.** The Save/Load screen reads save *manifests* first, never the full payload, and exposes them via `selectors.persistence.saveSlotManifests`. Each manifest powers the slot row (timestamp, scenario name, player, thumbnail, content-hash status, turn number) and the compatibility seal. Selecting a slot computes `selectors.persistence.selectedSaveCompatibility` (version / hash / migration result) — still without hydrating game state.

A full payload load (decompress + replay + hash-verify) only happens when the user confirms "Load," at which point the screen routes to the loading screen (`59`).

What is **not** specified is the on-disk separation between manifest and payload — whether the IndexedDB record stores them as one blob (in which case the manifest is "logical" only, requiring the full row to be fetched and parsed) or as separate columns/rows. This affects how cheap manifest-only listing actually is in practice, but does not change the inspection-without-replay contract.

**Evidence:**
- `docs/architecture/wiki/screens/55-save-load/data-contracts.md` (`selectors.persistence.saveSlotManifests`, `selectors.persistence.selectedSaveCompatibility`, "Reads save manifests first.")
- `docs/architecture/wiki/screens/55-save-load/spec.md` (slot table with timestamp / scenario / player / thumbnail / content-hash status; CompatibilitySeal)
- `docs/architecture/diagrams/25-load-flow.md` (full replay only on Load action)

### Q: 163. Are replays bounded in size, or do they grow unbounded?

**Status:** ❌ UNKNOWN

**Answer:**
**Effectively unbounded by design, with soft size targets.** The save model is "log forever from `seed`" — every command since scenario start is retained, and `Checkpoint[]` (log index + hash) is *optional* for faster seek but does not prune the log. There is no documented cap on:

- Maximum command-log length (commands count).
- Maximum compressed save size before the system rejects further saves.
- A log-rotation / log-truncation policy for very long campaigns.
- A snapshot-and-prune policy (e.g., "after N turns, freeze a verified state and start a fresh log").

Soft targets exist:
- `< 50 KB` (gzipped) for a 7-day game (`02-log-only-save-format.md`).
- `~50–200 KB` typical (`24-save-flow.md`).
- Load must reach the adventure map within 2 s for a 7-day save (`03-save-load-ui.md`).

For a **multi-month** campaign or a long multiplayer match, no design protects against log growth. The optional checkpoints help load latency by avoiding from-zero replay, but they do not bound size.

**Evidence:**
- `tasks/mvp/08-persistence/02-log-only-save-format.md` (`commandLog: Command[]` — full log; `checkpoints: Checkpoint[]` optional, for seek not for trimming; "< 50KB for a 7-day game")
- `docs/architecture/diagrams/24-save-flow.md` ("~50-200 KB compressed")
- `tasks/mvp/08-persistence/03-save-load-ui.md` (2 s load target for 7-day save)
- No log-pruning, snapshot-and-rebase, or size-cap rule in any task.

## 🔍 Summary

### Missing Logic
- **Migration scripts and policy** (Q155, Q156) — gates and UI states exist; no migrator format, support window, or deprecation timeline.
- **Autosave cadence and blocking model** (Q160) — option exposed in UI, slot reserved in mockup; no interval, no async-vs-sync rule, no retention policy, no failure handling, no multiplayer interaction.
- **Atomicity / crash-safety contract** (Q161) — no shadow-write / verify-then-swap policy; relies implicitly on IndexedDB transactions but not documented.
- **Compression determinism** (Q154) — gzip used, level/library not pinned.
- **Byte-identical save guarantee** (Q151) — canonical bytes are deterministic, but per-save metadata (`createdAt`, `id`, `name`) and unfixed compression breaks bit-equality of on-disk files.
- **Save format vs. replay format separation** (Q152) — the engine treats them as one; no envelope spec for shared replays / bug-report fixtures / spectator playback.
- **Replay size bounds and log-pruning policy** (Q163) — log grows linearly with command count; no rotation / snapshot-and-rebase / per-save size cap.
- **Save vs. payload split in storage** (Q162) — manifest selectors implied, but on-disk record layout (single blob vs. separated manifest/payload) is unspecified, affecting list-page cost.
- **Diagram drift** (Q150) — `24-save-flow.md` shows a `state` blob in the save file; the canonical task definition is log-only. One of these documents must be corrected.

### Risks
- **Single-source determinism risk:** Because saves are *only* command logs, any non-determinism bug (a stray `Date.now`, an unsorted iteration, a float in a formula, a content-pack edit between save and load) produces an unreplayable save and a hard "Save corrupt!" error rather than partial recovery. The fuzz harness (M1) is the only line of defence.
- **No log pruning:** A long campaign or extended multiplayer match could accumulate tens of thousands of commands, ballooning load time well past the 2 s target. The "checkpoints" feature mitigates load latency but does not cap size.
- **No autosave spec:** Players can lose hours if a crash occurs mid-turn and autosave is implemented naively (e.g., per-day on End-Day only, sync-blocking the UI).
- **Tampering is detected, not prevented:** xxh64 is non-keyed; a malicious modder can edit the log and re-hash. This is fine for single-player and matters little for cooperative multiplayer (where the per-turn state-hash exchange catches it), but for any future leaderboard / ranked / tournament mode, signed-manifest authentication will be required.
- **Pack-hash mismatch UX is "warn-or-abort":** That is fine for player-driven loads but not for autosaves resumed across pack updates — silently failing autosaves is unacceptable, but no policy is defined.
- **Atomic-write absence on export:** A tab crash mid-`exportSave` produces a truncated `*.hrsa.json` on the user's filesystem; load gates reject it, but a player who only kept that copy has lost the save.
- **IndexedDB quota:** Browsers cap IndexedDB at ~50 MB–unbounded depending on policy and platform. No quota-handling task exists.
- **MP / persistence interaction:** Whether saves include the in-flight multiplayer command log, who can save during a remote match, and how a save loaded into multiplayer reconciles with the remote peer's seed are all undocumented.

### Improvements
- **Author the migration registry now**: `src/persistence/migrations/v1-to-v2.ts` style; declare the support window (e.g., "we migrate any save ≥ N versions old; older saves are export-only"); ship a migrator template and an automated test that round-trips every supported version.
- **Define autosave**: cadence (every End-Day on the local turn boundary), blocking (write asynchronously after the command flush, never on the render path), retention (rotating 3 slots: `auto-1`, `auto-2`, `auto-3`), failure UX (silent retry once, then non-modal toast).
- **Pin gzip determinism**: choose `pako` at level 6 (or browser `CompressionStream`), document the choice in `02-log-only-save-format.md`, and add a test that re-saving an unchanged session produces identical bytes (after stripping the dynamic metadata header).
- **Separate manifest from payload on disk**: store the manifest as a separate IndexedDB record keyed by `${id}:manifest` so list views never read the full log; keep the payload at `${id}:payload`. Both written in one transaction.
- **Adopt write-temp-then-rename for export**: write to `${name}.tmp.json`, fsync (where supported), atomic rename to `${name}.hrsa.json` — or at minimum, finalize the blob and trigger download in a single tick rather than streaming.
- **Add a snapshot-and-rebase policy**: every K turns (or every M commands), capture a verified canonical state snapshot, rotate the active log, and store `(snapshot, log_since_snapshot)`; on load, hydrate from snapshot and replay only the tail. Cap save size at e.g. 1 MB compressed.
- **Document the save-format vs replay-format relationship explicitly**: even if they share fields, give replays an `intent: "replay"` discriminator and forbid metadata leakage (player name, save title) so shared replays are PII-clean.
- **Add a determinism CI gate for saves**: every save written by the test suite must reload, hash-verify, and produce a stable `stateHash` across machines.
- **Resolve the diagram drift in `24-save-flow.md`**: remove the `state` blob from the example payload; align with `02-log-only-save-format.md` and `25-load-flow.md`.
- **Quota policy**: detect IndexedDB quota errors, surface a "manage saves" CTA, and recommend export of older saves before allowing new ones.
- **Specify multiplayer + persistence**: e.g., "saving is host-only; loading restarts the local session and re-establishes lockstep from the saved log."
